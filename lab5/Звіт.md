### Мета роботи
Реалізувати в системі `FootballHub Connect` механізми автентифікації та авторизації користувачів, а також розмежування прав доступу до даних за допомогою `Supabase Auth` та `Row Level Security (RLS)`.

---

### 1. Модель користувача та безпека даних
* [cite_start]**Система ідентифікації:** Використовується вбудований сервіс `auth.users` від Supabase для керування обліковими записами[cite: 107].
* [cite_start]**Зберігання паролів:** Паролі хешуються на стороні сервера Supabase (алгоритм `bcrypt`) і не доступні у відкритому вигляді[cite: 108].
* [cite_start]**Прив’язка даних:** Кожен запис у таблиці `players` містить поле `user_id` (UUID), пов’язане з конкретним авторизованим користувачем[cite: 109].

---

### 2. Процес Автентифікації (Authentication)
[cite_start]Взаємодія з системою реалізована через компонент `Auth.jsx` та метод `supabase.auth.signInWithPassword`[cite: 111].
* [cite_start]**JWT (JSON Web Token):** При успішному вході клієнт отримує токен, який зберігається у `localStorage`[cite: 112].
* [cite_start]**Управління сесією:** У файлі `App.jsx` реалізовано слухач подій `onAuthStateChange`, який миттєво реагує на вхід або вихід користувача[cite: 113].

---

### 3. Авторизація (Authorization) та RLS
[cite_start]Захист даних реалізовано безпосередньо в базі даних через **Row Level Security (RLS)**[cite: 115]:
* [cite_start]**Політика SELECT:** Користувач бачить лише гравців, де `user_id` збігається з його ID[cite: 116].
* [cite_start]**Політика INSERT/UPDATE:** Дозволено лише для власних записів авторизованих користувачів[cite: 117].
* [cite_start]**Клієнтська перевірка:** Умова `if (!session) return <Auth />` блокує інтерфейс без активної сесії[cite: 118].

---

### 4. Реалізація функцій керування (CRUD)

| Дія (Маршрут Supabase) | Опис функціоналу | Доступ |
|:---|:---|:---|
| `SELECT (players)` | Отримання списку своєї команди | [cite_start]Авторизований Coach [cite: 121] |
| `INSERT (players)` | Додавання нового гравця | [cite_start]Авторизований Coach [cite: 121] |
| `UPDATE (players)` | Редагування даних або статусу | [cite_start]Авторизований Coach [cite: 121] |
| `DELETE (players)` | Видалення гравця з бази | [cite_start]Авторизований Coach [cite: 121] |

### 5. Тестування сценаріїв безпеки
* [cite_start]**Доступ без сесії:** Система автоматично перенаправляє на компонент `Auth`[cite: 123].
* [cite_start]**Валідація даних:** Перевірка обов'язкових полів (наприклад, `newNumber`) при додаванні[cite: 124].
* [cite_start]**Вихід із системи:** Метод `supabase.auth.signOut()` очищує сесію та повертає до екрана входу[cite: 125].
