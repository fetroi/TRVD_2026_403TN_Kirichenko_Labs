### 1. Що таке SPA (Single Page Application) та чим воно відрізняється від MPA?
**SPA** — це веб-додаток, який завантажує лише одну HTML-сторінку. Весь подальший вміст оновлюється динамічно за допомогою JavaScript без повного перезавантаження браузера.

| Характеристика | SPA (напр. React) | MPA (Класичні сайти) |
| :--- | :--- | :--- |
| **Перезавантаження** | Відсутнє (тільки зміна даних). | Повне при кожному переході. |
| **Швидкість** | Дуже швидка після завантаження. | Залежить від швидкості сервера. |
| **SEO** | Потребує додаткових налаштувань. | Відмінне "з коробки". |
| **Користувацький досвід** | Схожий на мобільний додаток. | Переривчастий (білий екран при переході). |

---

### 2. Що таке Virtual DOM і навіщо він потрібен?
**Virtual DOM** — це легка копія реального DOM-дерева у пам'яті (у вигляді об'єктів JavaScript).

**Навіщо він потрібен:**
Маніпуляції з реальним DOM у браузері дуже повільні. React спочатку вносить зміни у Virtual DOM, порівнює його з попередньою версією (процес Diffing) і оновлює в реальному браузері лише ті елементи, які дійсно змінилися. Це значно підвищує продуктивність інтерфейсу.

### 3. Як працює Routing на клієнті та роль History API?
При клієнтському роутингу браузер не робить запит на сервер за новою сторінкою. Замість цього JavaScript перехоплює натискання на посилання.

* **Звичайні посилання:** Змушують браузер очистити пам'ять і завантажити новий HTML-документ із сервера.
* **History API:** Дозволяє JS змінювати URL в адресному рядку (`pushState`) без перезавантаження сторінки. Це дозволяє користувачу використовувати кнопки "Назад" і "Вперед", зберігаючи при цьому стан SPA.

### 4. Де безпечніше зберігати JWT токени на клієнті?
1. **localStorage / sessionStorage:** Найпростіший варіант, але вразливий до **XSS-атак** (зловмисний скрипт може легко прочитати токен через JS).
2. **HttpOnly Cookies:** **Найбезпечніший варіант.** Таке cookie неможливо прочитати через JavaScript. Це захищає токен від викрадення через XSS. Браузер сам автоматично додає це cookie до кожного запиту на сервер.

### 5. Що таке CORS і як це вирішити?
**CORS (Cross-Origin Resource Sharing)** — це механізм безпеки браузера, який блокує запити з одного домену (напр. `localhost:3000`) до іншого (напр. `api.football.com`), якщо сервер явно не дозволив такий доступ.

**Як вирішити:**
Сервер повинен надіслати спеціальний заголовок `Access-Control-Allow-Origin: *` (або вказати конкретний домен фронтенду). У Supabase це налаштовується в розділі "API Settings -> CORS".

### 6. Концепція State Management (Локальний vs Глобальний стан)
**Управління станом** — це збереження та оновлення даних, які бачить користувач.

* **Локальний стан (useState):** Достатньо, якщо дані потрібні лише одному компоненту або його прямим нащадкам (наприклад, текст у полі вводу або стан відкриття одного випадаючого списку).
* **Глобальний стан (Context/Redux/Pinia):** Потрібен, коли дані мають бути доступні в різних частинах додатка (наприклад, дані про авторизованого тренера, загальний список гравців або обрана мова інтерфейсу). Це позбавляє від необхідності передавати дані через 10 рівнів компонентів (Prop Drilling).
