### 1. Порівняння REST, GraphQL та gRPC

| Технологія | Протокол | Формат даних | Типізація | Найкраще для: |
| :--- | :--- | :--- | :--- | :--- |
| **REST** | HTTP 1.1 | JSON / XML | Слабка | Публічних API, стандартних вебсервісів. |
| **GraphQL** | HTTP | JSON | Сувора | Складних мобільних та фронтенд-додатків. |
| **gRPC** | HTTP/2 | Protobuf (Binary) | Дуже сувора | Мікросервісів, високонавантажених систем. |

**Коли використовувати:**
* **REST:** якщо потрібна простота та широка підтримка браузерами.
* **GraphQL:** якщо клієнту потрібно запитувати лише конкретні поля (уникнення надлишкових даних).
* **gRPC:** для внутрішньої взаємодії між серверами, де важлива швидкість та мінімальний розмір пакетів.

---

### 2. Проблема Over-fetching та Under-fetching
* **Over-fetching (Надлишок):** Сервер повертає забагато даних (наприклад, весь профіль гравця, коли нам потрібне лише його прізвище).
* **Under-fetching (Дефіцит):** Один ендпоінт дає замало даних, тому клієнту доводиться робити 2-3 додаткові запити.

**Як GraphQL вирішує це:** Клієнт сам описує структуру запиту. Він каже: "Дай мені лише `name` та `number`", і сервер повертає рівно ці два поля. Це усуває обидві проблеми одним запитом.

### 3. Що таке Protocol Buffers (Protobuf) та його переваги?
**Protobuf** — це механізм серіалізації структурованих даних у бінарний формат, розроблений Google.

**Переваги перед JSON:**
1. **Розмір:** Бінарні дані значно менші за текстовий JSON.
2. **Швидкість:** Парсинг Protobuf відбувається в рази швидше.
3. **Типізація:** Структура даних чітко описана у `.proto` файлах, що виключає помилки несумісності.

---

### 4. Версіонування API
* **У REST:** Зазвичай через URL (`/v1/players`, `/v2/players`) або через заголовки (Headers). При змінах часто доводиться підтримувати дві версії коду паралельно.
* **У GraphQL:** Офіційна позиція — **"немає версіонування"**. Замість створення нової версії, старі поля позначаються як `@deprecated`, а нові просто додаються. Клієнти поступово переходять на нові поля без ламання старої логіки.

---

### 5. Концепція Idempotency (Ідемпотентність)
**Ідемпотентність** — це властивість операції, при якій повторний виклик того самого запиту з тими самими параметрами не змінює стан сервера більше одного разу.

**Методи:**
* **Ідемпотентні:** `GET` (просто читання), `PUT` (заміна даних), `DELETE` (видалення вже видаленого нічого не змінить).
* **НЕ ідемпотентні:** `POST` (кожен новий запит створить новий запис у базі даних).

---

### 6. Роль SDL у GraphQL та .proto у gRPC
Ці файли виконують роль **"контракту"** між клієнтом та сервером:

* **SDL (Schema Definition Language):** У GraphQL описує всі доступні типи даних, запити (Queries) та мутації. Клієнт завжди знає, які дані він може отримати.
* **.proto файли:** У gRPC описують структуру повідомлень та сервісні методи. На їх основі автоматично генерується код для різних мов програмування (Java, JS, Go, Python), що гарантує, що сервер і клієнт "говорять однією мовою".
