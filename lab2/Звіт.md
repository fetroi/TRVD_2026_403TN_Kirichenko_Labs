### 1. Обраний технологічний стек
* [cite_start]**Frontend:** `React.js (Vite)`[cite: 33]. [cite_start]Компонентний підхід дозволяє швидко створювати динамічні таблиці гравців та картки статистики[cite: 34].
* [cite_start]**BaaS (Backend as a Service):** `Supabase (PostgreSQL)`[cite: 35]. [cite_start]Надає готову автентифікацію, базу даних та автоматичне REST API «з коробки»[cite: 36].
* [cite_start]**Database:** `PostgreSQL`[cite: 37]. [cite_start]Надійна реляційна база для зберігання списків гравців, їхніх номерів та статусів[cite: 38].
* [cite_start]**Security:** `Supabase Auth + RLS` (Row Level Security)[cite: 39]. [cite_start]Дозволяє розмежовувати доступ до даних безпосередньо в базі[cite: 40].

---

### 2. Архітектурна модель (C4 Simplified)

#### 2.1 System Context (Контекст)
[cite_start]Користувач (Тренер) взаємодіє з веб-додатком для перегляду статистики, додавання гравців та зміни їхнього статусу здоров'я[cite: 43].

#### 2.2 Container Diagram (Контейнери)
* [cite_start]**Браузер (React SPA):** Відображає інтерфейс, обробляє логіку фільтрації та анімації[cite: 45].
* [cite_start]**Supabase Client (SDK):** Здійснює прямі запити до бази даних через захищений протокол[cite: 46].
* [cite_start]**Supabase Cloud:** Хмарне сховище, що відповідає за авторизацію та збереження таблиць[cite: 47].

### 4. Нормалізація
* [cite_start]**1НФ:** Кожен атрибут (номер, позиція) є атомарним[cite: 60].
* [cite_start]**2НФ:** Усі дані гравця залежать від його унікального первинного ключа[cite: 61].
* **3НФ:** Відсутні транзитивні залежності. [cite_start]Наприклад, статус здоров'я залежить безпосередньо від гравця, а не від його позиції[cite: 62].
