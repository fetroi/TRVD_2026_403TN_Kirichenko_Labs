### Мета роботи
Реалізувати в системі FootballHub Connect механізми автентифікації та авторизації користувачів, а також розмежування прав доступу до даних за допомогою Supabase Auth та Row Level Security (RLS).

---

### 1. Модель користувача та безпека даних
* **Система ідентифікації:** Використовується вбудований сервіс auth.users від Supabase для керування обліковими записами.
* **Зберігання паролів:** Паролі хешуються на стороні сервера Supabase (алгоритм bcrypt) і не доступні у відкритому вигляді навіть адміністратору.
* **Прив’язка даних:** Кожен запис у таблиці players містить поле user_id (UUID), яке пов’язане з конкретним авторизованим користувачем.

---

### 2. Процес Автентифікації (Authentication)
Взаємодія з системою реалізована через компонент Auth.jsx та метод supabase.auth.signInWithPassword.
* **JWT (JSON Web Token):** При успішному вході клієнт отримує токен, який автоматично зберігається у localStorage.
* **Управління сесією:** У файлі App.jsx реалізовано слухач подій onAuthStateChange, який миттєво реагує на вхід або вихід користувача, оновлюючи стан session.

---

### 3. Авторизація (Authorization) та RLS
Замість зовнішніх Middleware, захист даних реалізовано на рівні бази даних через Row Level Security (RLS):
* **Політика SELECT:** Користувач може бачити лише тих гравців, у яких user_id збігається з його власним ID.
* **Політика INSERT/UPDATE:** Додавання та редагування записів дозволено лише авторизованим користувачам для їхніх власних записів.
* **Клієнтська перевірка:** У коді App.jsx додано умову if (!session) return <Auth />, що блокує доступ до інтерфейсу керування без активної сесії.

---

### 4. Реалізація функцій керування (CRUD)
У межах роботи було впроваджено наступні операції, захищені авторизацією:

| Дія | Опис функціоналу | Доступ |
|:---|:---|:---|
| SELECT (players) | Отримання списку команди, закріпленої за тренером. | Авторизований Coach |
| INSERT (players) | Додавання нового гравця (номер, позиція) до бази. | Авторизований Coach |
| UPDATE (players) | Редагування номера, позиції або статусу здоров'я. | Авторизований Coach |
| DELETE (players) | Миттєве видалення гравця з бази даних. | Авторизований Coach |

---

### 5. Тестування сценаріїв безпеки
* **Доступ без сесії:** При спробі відкрити головну сторінку без входу, система автоматично перенаправляє на компонент Auth.
* **Валідація даних:** При додаванні гравця перевіряється наявність обов'язкових полів (newNumber).
* **Вихід із системи:** Метод supabase.auth.signOut() коректно очищує сесію та токени, повертаючи користувача до екрана входу.

---

### Висновок
У ЛР4 успішно інтегровано систему контролю доступу. Використання JWT-токенів у поєднанні з політиками RLS забезпечує високий рівень безпеки даних, де кожен тренер має доступ виключно до своєї команди.
